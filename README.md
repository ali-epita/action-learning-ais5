# AIS 5 — Small VLMs vs. GUI Specialists

An empirical study of efficiency-performance trade-offs for on-device screen agents.
EPITA Action Learning, Spring 2026.

> **Central question.** Under strict on-device budgets (≤4B parameters, ≤8 GB VRAM),
> can small general-purpose VLMs, suitably adapted, match specialized GUI models on
> screen-interaction tasks?

## Setup

### Local (uv)

```bash
make sync              # install deps and the package itself
make test              # run CPU tests (no GPU / network needed)
```

To bring in the heavier optional groups:

```bash
make sync-quant        # adds bitsandbytes / AWQ / GPTQ (Linux + CUDA only)
make sync-all          # everything optional
```

### Docker

```bash
make docker-build      # builds ais5:latest (CPU base, ~2 GB)
make jupyter           # → http://localhost:8888 (no token, local-only)
make docker-shell      # bash inside the dev container
make docker-down       # stop and clean up
```

The Hugging Face cache is a named volume (`hf-cache`) so model and dataset
downloads survive image rebuilds.

For CUDA, build with a different base:

```bash
docker build --build-arg PYTHON_VERSION=3.11 -t ais5:cuda .
# (and adjust the runtime stage to a CUDA image, e.g. nvidia/cuda:12.4-runtime-ubuntu22.04)
```

## Usage

### Quick smoke test

```bash
# 50-sample zero-shot eval of Qwen2.5-VL-3B on ScreenSpot-V2.
uv run python -m ais5.cli eval --config configs/eval/zero_shot_qwen.yaml --limit 50
```

### From a notebook

```python
from ais5.data import load_benchmark
from ais5.eval import evaluate_model
from ais5.models import get_model

model = get_model("qwen2.5-vl-3b")
samples = list(load_benchmark("screenspot-v2"))
run = evaluate_model(model, samples, limit=100)
print(f"accuracy = {run.accuracy:.4f}")
```

### CLI commands

```bash
make eval         # zero-shot Qwen2.5-VL → ScreenSpot-V2
make train        # LoRA fine-tune Qwen2.5-VL on OS-Atlas + UGround
make bench        # full Task 3 grid (model × quant × benchmark)
make papers       # (re-)download bibliography PDFs
```

## Design notes

- **Same prompt/action format across models.** All wrappers go through the
  unified `parse_click` in `ais5.prompt.action`, so the click-extraction logic
  is identical regardless of how the model formats its output.
- **Fair-comparison constraints in one place.** Every modeling task imports
  `evaluate_model` from `ais5.eval`. Don't re-implement click accuracy in
  notebooks — extend the runner if you need extra hooks.
- **8 GB on-device budget is enforced as a software cap.** Free Colab/Kaggle
  GPUs are 16 GB; we measure peak VRAM and reject configs that exceed 8 GB
  rather than depending on physically smaller hardware.
- **Heavy imports are lazy.** Importing `ais5.models` does not import
  `transformers` or `torch` until you actually instantiate a model. CPU-only
  notebooks (and the test suite) stay snappy.
